1 · Executive summary

If you read nothing else

The codebase is in remarkably good shape for something that's been vibe-coded across half a dozen models over a year. The domain model is clean (System → flat nodes[] → BodyFactory → SystemProcessor), the physics layer is properly separated from rendering, and there is a real baseline test that anchors Earth, Venus, Mars, Jupiter, Io against measured values. That's the load-bearing wall and it's standing.

The accumulated grime is mostly in three places: two enormous Svelte components carry most of the UI state, an entire time-control subsystem is copy-pasted between them, and the main orbital canvas has zero touch input. None of these are structural — they're each a focused refactor.

Three calls I'd make

1. Don't full-rewrite. Fork to beta, then run three sequenced refactors on the existing tree: extract shared modules, split the god components, and replace mouse handlers with Pointer Events. Keep the physics engine essentially untouched.

2. Publish a "Physics Honesty" page. Most of your deviations from reality are defensible. The problem is that nobody can find them unless they read ARCHITECTURE.md. Surface them in-app as a clickable footnote on every derived value, so a GM can answer "why is Jupiter 165 K?" without you in the room.

3. Design the UI around two surfaces, not three. Desktop and tablet share one layout that gracefully degrades; phone gets a focused subset of that same layout. Don't try to make TransitPlanner work on a 375 px screen — show a "Use a larger device for flight planning" affordance and let the rest of the app run.

2 · Architecture & duplication

Where the rot is, where it isn't

2.1 What's working well

  • Layered pipeline: generation/ builds structure, physics/ is pure functions, core/SystemProcessor orchestrates, components/ renders. The contract between layers is clear and respected.
  • Flat nodes[] with parentId beats a nested tree for serialisation, undo, and hierarchy edits — good call. The BodyFactory as the single instantiation point keeps shapes consistent.
  • BigInt time in temporal/ is genuinely the right tool for galactic timescales. The dual-clock (Master vs Display) decouples mission state from UI scrubbing — re-use this pattern in the v2 toolbar.
  • Floating-origin renderer (SystemVisualizer.svelte:render) is the right move — without it, you get the well-known 32-bit jitter when you zoom into a moon while the camera transform still carries the full AU offset.
  • Vitest baseline in lib/system/physics-baseline.test.ts is the single most valuable file in the repo. Earth/Mars/Venus/Jupiter targets are assertions, not aspirations. Protect this with your life.

2.2 The god components

Three files dominate the component layer. They're not equally bad, but they're all past the point where new behaviour can be added without anxiety.

2.4 Recommended file-layer changes

Beyond de-duping, these structural moves pay off the moment v2 work starts:

  • Split SystemView into a layout shell + concern slots. Pull the TimePanel, FocusHeader, BodyDetailsPane, ConstructDetailsPane, PlannerPane and ShipLogPane into sibling components. SystemView becomes ~400 lines of state glue + a layout grid.
  • Promote a lib/viewport/ module. Camera state (panStore, zoomStore, follow vs manual mode, auto-zoom damping) currently lives partly in cameraStore.ts and partly inline in SystemVisualizer. Pull it together — it's the API the new touch gesture layer needs to talk to.
  • Introduce lib/input/. A single useGestures(el, { onPan, onZoom, onTap, onLongPress }) Svelte action that abstracts over PointerEvents. Both SystemVisualizer and Starmap become consumers, and @panzoom/panzoom can go.
  • Co-locate report markup with its data. routes/report/+page.svelte is 2 900 lines because it inlines every section. Move the per-section markup to lib/reports/sections/*.svelte and keep the route as a thin print-mode shell.

3 · Physics audit

Plausible > accurate > cinematic — where each system sits

You asked for "plausible, with every deviation defensible." That's the lens I'm grading against. Each row below is one physical system, with a one-line summary, the verdict (defensible, workable but ought to be documented, or actually wrong), and a recommended action.

I've also flagged where you can credibly push toward more accuracy — n-body in particular is more tractable in a browser than you may think, given the timescales you actually care about.

System What you do How it differs from reality Verdict
Kepler propagation
physics/orbits.ts:propagateState 		Newton iteration on M = E − e sin E with a bisection fallback for high e. Returns position and velocity in AU, AU/s. 2-D only (i_deg stored but ignored at render time), no relativistic precession. Both are fine for TTRPG scope. defensible
M ≤ 30 iter, then bisect
Multi-body forces during transit
transit/math.ts:integrateBallisticPath 		RK4 integrator that already accepts an optional N-body gravity list (nBodyNodes). Currently only the primary host is passed in by callers. The mechanism is there; the call sites only sum a single Mu. So your "N-body summation" claim in v1.9.2 is half-true. document or wire up
3-body sampling is cheap
Lambert solver
transit/math.ts:solveLambert 		Universal-variable (Vallado-style) with 200-iter bisection on z. Bisection is slow and can stall near multi-rev boundaries; standard practice is Newton-Raphson on F(z). Currently masked by the 10 000 km/s "insanity cap." swap to Newton
~50 LOC change
Stellar mass–luminosity
physics/stellar-evolution.ts:deriveStarFromHR 		Falls back to M/M☉ = L^0.28 (i.e. L ∝ M^3.5). That's the canonical mid-MS relation. Applied unconditionally — including to giants/supergiants/sub-dwarfs where it's wrong by 1–2 orders of magnitude. branch by lumClass
Stellar evolution / ageStar
physics/stellar-evolution.ts:ageStar 		L(t)/L0 = 1 + 0.8·(age/τ_MS) on the main sequence; +500× at red-giant peak; remnant L,T are constants. For a G-star, +80 % over the full MS is in the right ballpark (Sun started at ~70 % L☉). +500× peak is high for a low-mass star (more like 1 000–2 500 L☉ for a solar-mass tip-of-RGB), but the right shape. tune coefficients per mass
Equilibrium temperature
physics/temperature.ts:calculateEquilibriumTemperature 		Multi-star flux summation via LCA-walked distances. Bond albedo estimated from composition. Uses the time-averaged (mean) orbital distance, not the rms-flux distance for eccentric orbits. You have the right helper (equivalentFluxDistanceAU) — it's just not threaded into the equilibrium calc. tiny fix, real upside
Greenhouse forcing
physics/atmosphere.ts:calculateGreenhouseEffect 		Hybrid √P · log(1 + √(100 pp)) per gas, with CIA boost, cryogenic spectral-overlap attenuation, and a dense-CO₂ Venus boost. Saturating final response. Composite empirical fit — neither RRTM nor radiative-convective. Calibrated to hit Earth ≈ +33 K, Venus ≈ +500 K, Mars ≈ +5 K in the baseline test. defensible (fit, not first-principles)
Gas-giant "surface" T Above 1 000 bar, effective pressure for greenhouse is capped at 1 bar. Reports cloud-top T. The right choice for GM intuition. ARCHITECTURE.md explains why — surface this in the tooltip. defensible — surface the why
Habitable zone
physics/zones.ts:calculateGoldilocksZone 		Kopparapu et al. conservative HZ (runaway-greenhouse → maximum-greenhouse), with binary-companion S_eff coupling. Properly implemented. Companion coupling solves S = L₁/r² + L₂/(r²+s²) as a quadratic — clean. solid
Frost / silicate / soot lines Inverted blackbody at fixed T (170 K formation, 125 K current, 70 K CO₂, 30 K CO). ZAMS back-calc by spectral-class α. Correct in shape. The α values are 70–80 % luminosity gain over current age for a G star — defensible (real Sun: ~+30 % since formation, but the input "age" is total MS lifetime here, not 4.6 Gyr). defensible
Orbital stability
physics/stability.ts 		Adjacent-pair scan with mutual-Hill spacing (Δ < 3.5 critical, < 5.5 tight, < 8.5 marginal) and Hill-sphere boundary checks for the host. Δ ≈ 3.46 = 2√3 is the literature cutoff for two-planet Lagrange stability, so 3.5/5.5/8.5 maps onto Hill-stable / Lagrange-stable / packed. Only weakness: it ignores resonances. defensible
Surface radiation
physics/radiation.ts 		Splits stellar flux into 90 % photons / 10 % particles. Magnetosphere shields particles only; atmosphere shields both via per-gas shielding coefficient × Beer–Lambert. Right shape, wrong split for high-activity stars (M dwarfs emit relatively more in particles). 90/10 is a Sun-like-only fit. Earth lands at 2.0–3.0 mSv/yr in the baseline test, which is correct. make split depend on spectral class
Roche limit Rigid-body limit with satellite density = 3000 kg/m³ assumed. Rigid is the conservative branch (real bodies disrupt sooner — fluid limit). 3000 kg/m³ is a rock assumption that under-predicts disruption for icy moons. document assumption; sat density from comp.
Atmospheric scale height & retention Isothermal H = RT/(g·M). Jeans escape ratio v_e/v_th > 6 stable, > 4 unstable. Both are textbook. The Jeans-only model misses hydrodynamic escape (matters for early M dwarfs zapping atmospheres off close-in planets) — could be flagged as a tag rather than modelled. defensible
Tidal heating Captured as tidalHeatK delta with tidal/hotspots tag, calibrated so Io ≈ +4–5 K mean. Real Io is +130 K vs equilibrium (because most heat is from a few hotspots). Treating it as a global delta & flagging hotspots separately is the right GM-tool compromise. defensible
Δv budgets
orbits.ts:calculateDeltaVBudgets 		v_orbit + gravity loss (0.15 v_orbit + log10(P) bonus) + drag loss (1300 · P^0.6 m/s). Calibrated against Earth ≈ 9.3 km/s and Venus ≈ 27 km/s — both in the right ballpark. Aerobrake landing uses exp(−0.5 P) which approaches zero correctly but the constants are eyeballed. defensible
Lagrange points
orbits.ts:getOrbitOptions 		L1 ≈ 0.8 · Hill from secondary, L2 ≈ 1.2 · Hill from secondary, L3/L4/L5 at the secondary's orbit radius. The Hill-radius approximations are within ~3 % of the true Sun–Earth values for small-mass-ratio systems. L3 actually sits at ~1 + 7m/12M from the primary (i.e. slightly outside) — but for GM use, the offset is invisible. defensible
N-body (evolutionary engine, beta)
stellar-evolution.ts:stepNBody 		RK4, swept-sphere collision, ejection by energy + radial-velocity. Cluster radius default 50 AU. RK4 is symplectic-violating but for short cluster simulations (Stellar Dance phase) it's fine. Browser handles a few stars at 60 fps trivially. defensible; could move to leapfrog later

3.1 The four numbers I'd actually fix

  1. Use rms-flux distance for eccentric equilibrium temperature. equivalentFluxDistanceAU(a, e) = a · (1−e²)^¼ already exists in zones.ts — just wire it into calculateEquilibriumTemperature. Eccentric planets currently report a temperature that's slightly too cold.
  2. Branch the mass-luminosity fallback by luminosity class. Inverting L ∝ M^3.5 is fine for V (MS). For I/II/III, mass is much lower than the relation predicts (a 1000 L☉ red giant might be 1 M☉, not 7 M☉). Add a small table: V → 0.286 exponent, IV → 0.30, III → clamp to progenitor mass, I/II → ditto, VII → use IFMR.
  3. Switch Lambert to Newton-Raphson. Bisection is fine but slow and brittle near domain transitions. Newton needs F(z) = (x³·S(z) + A√y)/√μ − Δt and F'(z), both well-known. ~3× faster, more robust at high energy.
  4. Drop the photon/particle split to a spectral-class function. 90/10 for G; ~80/20 for M-dwarfs; ~95/5 for late-K. Three lines.

3.2 "Should we go full n-body?" — yes, conditionally

You asked. Yes, you can. The constraints are:

  • Don't replace Kepler for the visualiser. Kepler propagation is exact, deterministic, and free. Switching the main view to numerical integration would burn frame budget and accumulate error across long simulations.
  • Do switch transit ballistics to n-body summation. The integrator already supports it (nBodyNodes parameter, unused). Wire in every body whose Mu/r² is > 10⁻⁶ of the primary's at the path's midpoint. This is the difference between "your Hohmann from Earth to Mars goes straight through Jupiter's Hill sphere unperturbed" (current) and "it gets a real swing-by." The Trajectory Correction Manoeuvre machinery you already added in v1.9.2 makes this safe — drifts get absorbed as TCMs.
  • Use n-body only for Stellar Dance & v2 protoplanetary accretion. Both are already there. Don't expand the use of full n-body into long-term planetary evolution unless you switch to a symplectic integrator (Wisdom-Holman or REBOUND-style mercurius) — RK4 over Gyr will eat your energy.

Honesty page proposal

Build routes/physics/+page.svelte — a single-page appendix that quotes every coefficient with its derivation, every cap with its justification, and every test with its target. The rows from this audit are 80 % of that page already. Link from every derived-value tooltip in the main UI ("Why is this number what it is? →").

4 · The touch problem today

What works, what doesn't, what's actively broken

I went looking for touch handling across the whole tree. Here's the honest summary.

What works on a tablet right now

  • The Starmap. It uses @panzoom/panzoom with touch-action: none, so single-finger pan and two-finger pinch already work on the SVG.
  • Most modals — they're plain HTML forms, taps register fine, inputs come up the on-screen keyboard.
  • The time scrubbers — there's a real touchend handler on the range slider.

What doesn't

  • The main orbital canvas (SystemVisualizer). Zero touch events. mousedown / mousemove / mouseup / wheel only. You can tap to select but you can't pan, can't pinch-zoom, can't long-press to context-menu. This is the entire app for tablet users.
  • Hover-only affordances. :hover states are used everywhere — including for the "what does this button do" tooltips. Tablets see nothing on first touch and trigger the action on second touch.
  • Right-click context menus. No mobile equivalent. "Add System" on the Starmap requires a right-click.
  • Hit targets. The header time control buttons are 24 px tall × variable width. Apple HIG is 44 pt, Material Design is 48 dp.
  • Responsive layout. Exactly one @media query in the entire codebase (NewStarmapModal, < 768 px). Everything else is fixed grids and absolute pixel widths.

4.1 What I'd not change

  • Side panels. The current BodySidePanel / ConstructSidePanel with their tabbed sub-panels (Basics / Orbit / Atmosphere / Hydrosphere / Biosphere / Tags) are actually pretty good — they collapse cleanly, the tab metaphor scales to a phone if we restyle the chrome. Keep the structure, change the container.
  • The information architecture. The mental model is sound: Starmap → System → Body → Tab. Don't reinvent it; just stop making it depend on hover and right-click.

5 · v2 UI: principles & references

What good touch maps for spatial apps look like in 2026

5.1 Principles

  1. Pointer Events, not mouse events. Modern browsers unify mouse, touch, and pen behind PointerEvent. One handler set covers all three. Phase out on:mousedown/on:mousemove in favour of a useGestures Svelte action.
  2. Every action has a primary input. If you can right-click it, you can also long-press it. If you can hover for a tooltip, you can also tap an info chevron. No action requires a modifier the user doesn't physically have.
  3. Direct manipulation over chrome. Touch users want to drag the thing, not push a button next to the thing. Where a slider can become a draggable handle on the visualisation (orbital radius, eccentricity), prefer that.
  4. One canvas, one truth. Don't have a separate "tablet view." Make the canvas adapt to the input modality — the layout shell decides whether side panels become bottom sheets; the canvas behaves the same way underneath.
  5. Coarse-pointer = explicit affordances. CSS @media (pointer: coarse) lets us serve a different button density to touch users regardless of viewport size, which is the right axis. A user on a touchscreen laptop gets the touch affordances even at 1920 px wide.

5.2 References worth stealing from

You said you don't follow mobile patterns — here's the short list of apps that solve the same spatial-canvas-plus-data-panel problem well, and what's worth stealing from each.

Stellarium Mobile

One-finger drag = pan sky; pinch = zoom; tap = select object; drag selection up = pin info panel. The "tap to dismiss panel, tap object to swap target" affordance is the single best interaction you can copy for selecting bodies.

Apple Maps / Google Maps

The bottom sheet with three snap points (peek 80 px / half 50 % / full) is the right home for BodyTechnicalDetails on touch. Drag handle at the top. Content scrolls only when the sheet is at full snap. Apple Maps' "tap a label, sheet animates up to half-snap" is exactly the system-view → body interaction.

Procreate / Figma mobile

Floating, draggable tool palettes that snap to the screen edges and collapse to a single icon when not in focus. Excellent model for the visibility toggles (Names, Zones, L-points, Sensors) that currently sit in a button row at the top.

SkySafari / Solar Walk 2

The "time control as a radial dial" pattern (drag a finger around a wheel to scrub time) handles huge dynamic ranges (seconds → millennia) better than a slider. Optional — start with a redesigned slider, escalate to a wheel only if scrubbing ergonomics still feel cramped.

iA Writer / Bear (panels)

Both apps use an L-shaped layout on tablet: persistent narrow rail of icons on the left, expandable detail column, primary canvas on the right. Your existing System View collapses neatly into this — and unlike "everything's a modal", a rail stays out of the way of the canvas.

Universe Sandbox & Kerbal Map View

Hover-free body labels: persistent name tags appear only for the focused body and its immediate parent/children; tap a label to focus. This is what your "Show Names" toggle should do implicitly — it's noisy on touch.

6 · Proposed information architecture

Same model, two layout regimes, one input layer

The data, the panels, the modals — all good. The thing that changes is the shell they live in. Two layout regimes, same content; the shell picks one based on (pointer: coarse) and (max-width: ...).

6.1 Desktop / tablet shell (≥ 900 px wide)

Three-column grid. Left rail is permanent and narrow (icons + tooltip-on-touch). Centre is the canvas. Right column is the detail panel, collapsible to its header. The time-control strip is a fixed band along the top of the canvas, not the document.

Sketch · desktop/tablet shell — three-column with persistent left rail, collapsible right panel

6.2 Phone shell (< 900 px wide, coarse pointer)

Full-bleed canvas. Floating action bar bottom-right (primary verbs: Add, Focus, Reset, Toggle Labels). Bottom sheet replaces the right panel; default snap = peek (just the title), swipe up to half, swipe again to full. Time controls collapse into a single compact band pinned to the bottom of the canvas above the sheet.

Sketch · phone shell — full-bleed canvas, peek/half/full bottom sheet, FAB cluster

6.3 Input layer

One Svelte action, use:gestures, normalises all input. The visualiser knows nothing about mouse vs. touch. 

// usage on the canvas
<canvas
  use:gestures={{
    onPan:  (dx, dy)    => panBy(dx, dy),
    onZoom: (factor, x, y) => zoomAt(factor, x, y),
    onTap:  (x, y)      => pickAndFocus(x, y),
    onLongPress: (x, y) => openContextMenu(x, y),
  }}
  on:focuschange={...}
/>

Built on PointerEvent with two-pointer pinch tracking, a 150 ms long-press timer, and a 10 px movement threshold to distinguish tap from pan. Right-click still maps to long-press for keyboard-and-mouse users. No external library needed.

8 · Decisions I need from you

Open questions before kicking off Claude Code

  1. Phone scope. Confirm phone is "everything except Transit Planner, AI Expansion, and Evolutionary Wizard." Those three are too data-dense for a 375 px screen and not worth the layout cost. Phones get a clean read-only-ish browse + edit experience.
  2. Gesture lib vs. roll-our-own. I recommend rolling our own (≤ 200 LOC, no dep, full control). The alternatives are @use-gesture/vanilla (~5 kB, well-tested) or keeping @panzoom/panzoom for the Starmap and writing only the canvas handler. Pick a lane.
  3. Tooltip migration policy. Tooltips on title="..." attributes are everywhere. Do we migrate them eagerly (drop title, add Popover everywhere — touches every component) or lazily (keep title for desktop; only Popover where the content is non-trivial)?
  4. Right panel default on tablet. Open or collapsed by default? I lean collapsed so the canvas is the hero; open it on first selection. Confirm.
  5. Time scrubber: slider or wheel? Phase 3 ships a better slider. Wheel is a phase-5 upgrade if scrubbing still feels cramped on phone.
  6. Beta vs. main parity. Are we OK with beta.starsystemx.com diverging visually from main while Phase 3 ships, or do we hold Phase 3 behind a feature flag and merge to main first?
