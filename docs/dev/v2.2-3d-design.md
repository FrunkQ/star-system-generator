# V2.2 Paper Design — Theme: "3D"

Status: REVIEWED 2026-07-08 — all major calls decided (see the Decisions log at the end).
Feasibility + spec only; no code. Two workstreams:

- **Part A — Holo view**: a 3D rotatable hologram presentation of a system for the player-facing
  Field Guide. The GM view stays 2D and functional.
- **Part B — Photoreal window**: drive an external photoreal renderer (SpaceEngine / Universe
  Sandbox / alternatives) to give a "window from a construct" — look around and zoom from a
  spacecraft's vantage point.

Open questions for both parts are gathered in the final section.

---

## Part A — Holo view (Field Guide 3D tier)

### A1. Vision

A "starship bridge hologram" of the current system: the orbital data projected onto a 3D
rotatable grid, viewed from a shallow tilt. Planets and moons are lit, rotating oblate-spheroid
meshes with procedurally generated textures; the star is a camera-facing billboard glow (the
reference — Solaris by blyghtmare — renders its central star as exactly that: a 2D glow image
that always faces the camera). Holo-styled chrome, glow, scanlines, slight flicker. Slick and
refined; players poke around, rotate, zoom, tap bodies for facts, and watch worlds turn.

Explicitly NOT the GM view. The main UI keeps its 2D orrery. This is a new tier of the existing
Field Guide (`/catalogue?sid=`).

### A2. Reference analysis (Solaris, blyghtmare.itch.io/solaris)

Findings from inspecting the actual build and screenshots:

- It is a free browser tool for TTRPG worldbuilders, released mid-2026. Not a game.
- **It is not WebGL.** The entire tool is a single ~73 KB `index.html` using Canvas 2D with a
  hand-rolled perspective projection. The "hologram" is drawn ellipses and lines.
- Visual anatomy: near-black background with a dark teal tint; everything in ONE hue (bright cyan
  ~`#35fff6`) at varying alpha. A perspective-tilted square-cell grid plane fading at the edges.
  Dashed elliptical orbit rings. Wireframe-cage planets (3-6 lat/long ellipses). The star is a big
  soft radial-gradient billboard with faint lens spokes. Small monospace labels float above bodies.
  Heavy glow on everything; scanlines in holo mode; five selectable hologram colour schemes.

Two useful conclusions: (1) the look is cheap — additive glow + one hue + a fading grid carries
most of it; (2) we can exceed the reference easily (free-form rotation, depth sorting, real
textures from our procedural discs) by going WebGL.

### A4. Rendering stack decision

Three candid options, researched July 2026:

1. **Canvas 2D, hand-rolled projection** (what Solaris itself does). Zero dependencies, reuses our
   canvas experience. Fixed-tilt + azimuth rotation is easy; free rotation, depth sorting and glow
   quality get painful. Escape hatch, not the plan.
2. **three.js, dynamically imported on the holo route only** — RECOMMENDED. `three` core +
   OrbitControls + label projection lands ~150-185 KB gzip, paid ONLY when a player opens the holo
   tier (SvelteKit/Vite code-splits the `await import()` automatically; the 2D app pays zero
   bytes). Buys camera controls with touch + damping, sprites, depth, shaders, future headroom.
   Tree-shaking three is poor (renderer core pulls most of it) — accept the ~180 KB, don't fight it.
3. **Raw WebGL2** — ~10-15 KB payload, 2-4 focused days for billboards/lines/grid, but bloom,
   camera damping and maintenance are all on us. Only if the 180 KB is unacceptable.

Rejected: Babylon.js (~1.6 MB gzip core — strictly worse here); Threlte (Svelte wrapper for three,
mature on Svelte 5, but adds a layer for no gain when the scene is ONE self-contained imperative
module; revisit only if 3D spreads across many components).

### A5. Scene spec (the mobile-safe holo recipe)

Everything below runs without post-processing and holds 60 fps on phones:

- **Grid plane**: one quad + fragment shader — anti-aliased grid lines (`fwidth`-based) with
  radial fade to nothing. Polar grid (rings + spokes) suits orbital data; Solaris uses square
  cells. Make it a per-skin choice, default polar. One draw call.
- **Orbit rings**: anti-aliased shader annulus per orbit (quad + `smoothstep` ring, dash pattern
  from `atan`), tinted per body from existing apparent colour, brightness up when the body is
  selected. Resolution-independent and fat-line-safe (WebGL ignores `linewidth`). Each ring quad
  is oriented into its ORBITAL plane (rotate by Ω, i) — inclined orbits tilt out of the grid,
  which is most of the 3D payoff.
- **Bodies — lit oblate spheroid meshes, NOT billboards** (decision upgraded on review: billboards
  cannot rotate without re-rasterising the SVG per frame or pre-baking sprite sheets — both
  strictly worse than a one-time texture on a sphere the GPU animates for free):
  - Geometry: `SphereGeometry` scaled by the existing `oblatePolarFactor` = oblate spheroid;
    the spun-past-breakup case becomes an honest `TorusGeometry` (simpler than the SVG's
    tilted-annulus fake). ~3k triangles per body; 50 bodies ≈ 150k tris — trivial on any phone.
  - Texture: new `getPlanetTextureEquirect` — an equirectangular sibling of
    `getPlanetTexture` (`rendering/planetTexture.ts`), consuming the SAME inputs (apparent-colour
    palette roles, banding count, coverage fractions, per-body-id seeded PRNG, cache). Giants get
    EASIER (bands = horizontal stripes); rocky blob-scatter needs x-wrap for the seam (draw each
    blob twice at x and x±W). ~512×256 px planets, 256×128 moons (~0.5 MB GPU each).
  - Lighting: a point/directional light at the star's position → a REAL terminator that tracks
    the orbit automatically, and handles binary systems with two lights — both impossible for the
    SVG's baked gradient. Night side comes free.
  - Rotation: driven by `rotationPeriodSeconds` + `axial_tilt_deg` (all in the data); tidally
    locked bodies lock to the star naturally — eyeball worlds finally LOOK like eyeball worlds.
  - Clouds: second slightly-larger alpha-textured sphere at a different rotation rate.
  - Shadows: ring shadows + moon eclipses via one shadow-mapped star light — desktop-first
    (mobile fallback: the existing analytic ring-umbra maths in-shader, or none).
  - LOD: below ~8 px on screen, swap sphere → static impostor sprite (rasterised PlanetDisc) so
    zoomed-out systems stay cheap. Spheres use normal blending + depthWrite (they are solid
    objects); the additive-glow treatment applies to everything AROUND them.
  - Star: stays a soft radial glow BILLBOARD from `starColorFromTempK` + faint lens spokes — it
    is the light source, and it is the Solaris signature trick, verbatim.
  - Belts/rings: a dashed shader annulus band (reuse the orbit-ring shader with width + noise);
    planetary ring systems = alpha-textured annulus mesh tilted with the body.
  - Constructs: small glyph billboards (existing construct iconography), optional trajectory
    polyline from the active journey.
- **Drop-stems**: a thin vertical line from each body down to the grid plane. Classic hologram
  cue — and with real inclinations it is FUNCTIONAL, not decorative: stem length reads height
  above/below the reference plane (style above vs below differently, e.g. solid vs dashed).
  Cheap line segments.
- **Labels**: keep HTML. Project world → screen each frame (`vector.project(camera)` → CSS
  transform) and position existing Svelte label/tooltip components in an overlay div
  (`pointer-events: none`, re-enabled per label). No text in WebGL, no font atlases, existing
  styling. (three's CSS2DRenderer exists but re-parents DOM nodes — manual projection composes
  better with Svelte.)
- **Camera**: OrbitControls — rotate (azimuth free, polar clamped ~10-80° so players can't go
  under the table or fully top-down), pinch/wheel zoom clamped, damping on, no pan in v1 (focus
  follows selection instead).
- **GM highlight ping (DECIDED)**: the GM's focused body pulses/brightens in the player holo
  ("look at this"), riding the existing broadcast. If the pinged body is off-screen, the camera
  auto-rotates (damped tween) to centre-frame it, then hands control straight back to the player.
  Player camera is otherwise always free. Phase 2.
- **Skins (DECIDED: everything per-skin, all combinations possible)**: a skin is a parameter set
  `{ tint: cyan | green | amber | red | none(true-colour), gridMode: polar | square,
  bodyColouring: tinted | true-colour }`. Tint is a shader uniform and the grid mode a shader
  flag, so every scheme is nearly free; ship cyan/green/amber/red tinted skins AND a true-colour
  skin ("Expanse console": true-colour worlds inside tinted chrome). Defaults per skin chosen at
  build time; player picks skins in the existing catalogue skin switcher.
- **Scanlines + flicker**: CRTOverlay (CSS) over the canvas + a global opacity/emissive uniform
  modulated by `sin(t*a)*sin(t*b)` with occasional dropouts. Both effectively free.
- **Bloom**: NONE on mobile — additive blending over black with pre-softened textures reads as
  bloom already. Optional `UnrealBloomPass` gated to desktop (pointer: fine + high dpr) as a
  polish pass, half-res composer.
- **Scale handling**: Toytown-style log radial compression as a toggle (default ON, matching the
  orrery's readability trade-off) — in 3D, scale the full position vector so vertical offsets
  compress consistently with radius. Multi-star/barycentre systems come free via shared
  propagation (parent-chain accumulation is the same walk with 3D vectors).
- **Vertical axis — real 3D from day one**: the dataset carries full elements (`i_deg`,
  `Omega_deg`) that the 2D projection deliberately ignores; the holo view applies them via
  `propagateState3D`. The grid is the reference plane; inclined orbits tilt out of it; belts stay
  in-plane. KNOWN DIVERGENCE, accepted: the 2D orrery rotates by ω only, so for an orbit with
  non-zero Ω the body's azimuth in the holo view will differ slightly from the orrery, and the
  true top-down track of an inclined orbit is a node-line-squashed ellipse rather than the
  orrery's circle. The orrery's stylised flat view stays untouched (see open questions).

### A6. Integration + phasing

Placement (DECIDED): new catalogue theme `holo` (hi-tech tier sibling of `console`) AND a
projector mode from day one — same component behind both routes, different chrome. Starmap-level
3D ("Galaxy Overview" analogue) is IN V2.2 scope.

- **Phase 1 (core)**: extract `calculateWorldPositions` (+ toytown scaling) to a shared module
  and add `propagateState3D` (shares the Kepler solve with `propagateState`; applies the full
  Rz(Ω)·Rx(i)·Rz(ω) rotation — 2D behaviour byte-identical, untouched); new `src/lib/holo/` scene
  module (lazy three import); grid + orbit rings (plane-oriented) + star glow billboard +
  spheroid bodies (`getPlanetTextureEquirect` generator, star-light terminator, rotation + tilt,
  LOD impostor swap) + drop-stems + HTML labels; OrbitControls; holo tint; local time animation;
  catalogue theme wiring + skin switcher entry.
- **Phase 2 (guide integration)**: tap-to-select → bodyFacts panel; constructs + trajectories;
  belts + planetary ring meshes; cloud-layer spheres; the full skin set (tinted + true-colour);
  GM highlight ping with off-screen centre-framing; projector-mode chrome; flicker + scanlines;
  focus/zoom-to-body transitions.
- **Phase 3 (galaxy level — still V2.2)**: starmap-level 3D (systems as glow sprites on the grid
  with route lines — the "Galaxy Overview" analogue, reusing the same scene module); fly-in
  transition from starmap into a system; desktop bloom; shadow-mapped ring shadows + moon
  eclipses (desktop).

Estimated effort: Phase 1 is the bulk and is very tractable — the data, redaction, transport,
imagery and facts UI all exist; the new work is one WebGL scene module plus an extraction refactor.

### A7. Risks

- **Equirect texture generator scope creep**: `getPlanetTextureEquirect` must cover every body
  archetype the disc generator covers (giants, rocky, hot, ice, eyeball, cratered). Mitigation:
  same inputs and same seeded-PRNG structure as `getPlanetTexture` — port archetype by archetype,
  giants first (they get simpler in equirect); fall back to the LOD impostor sprite for any
  archetype not yet ported.
- **Texture seams/poles**: horizontal wrap needs blob duplication at x±W; pole pinch is
  acceptable for stylised worlds (real games ship it) — do not reach for cube maps in v1.
- **LOD impostor rasterisation** (PlanetDisc SVG → sprite): pure SVG (no foreignObject) so
  `img.src = svg blob` → canvas is safe cross-browser; cache aggressively; watch iOS
  texture-size limits (cap 256 px).
- **WebGL context loss** (mobile tab switching): handle `webglcontextlost/restored`, rebuild
  textures from the cache.
- **Old/weak devices**: feature-detect WebGL2 → fall back to the console (2D) tier with a notice.
- **Bundle discipline**: three must never leak into the main chunks — enforce via dynamic import
  only, verify in build output.

## Part B — Photoreal window from a construct

### B1. Vision and framing

A construct has a known position (and journey) inside a generated system. Give the table a
photoreal "window" from that vantage point — look around, zoom — by driving an external renderer,
rather than trying to render photoreal in the browser.

Framing matters: none of the candidate renderers can be embedded in a web page, so the "window" is
a second application window (or a second screen — which suits the existing projector/table-screen
story). And because every candidate is a desktop install, this is naturally a GM-machine/table
feature first, not a per-player-phone feature.

Three fidelity tiers fell out of the research:

- **Tier 0 — Export**: write the system out in the renderer's format; the user opens it manually
  and flies themselves. (We already have deep format knowledge from the IMPORTERS; the exporters
  are new but well-understood work.)
- **Tier 1 — Vista launch**: export PLUS a generated script that boots the renderer directly into
  the construct's exact position and time, UI hidden, free-look enabled. One click → one launch →
  the window. No live link.
- **Tier 2 — Live window**: a running renderer whose camera is continuously re-posed by SSG as the
  clock scrubs and the construct moves. Requires a real remote-control surface.

### B3. Decided shape (review of 2026-07-08)

**Phase B-1 (V2.2): SpaceEngine vista export — DECIDED, plus `.sc` export ships as a general
File > Export feature** (the vista is then just export + camera script; "generate in SSG, fly it
in SpaceEngine" is a first-class round-trip).

- New exporter `src/lib/export/spaceengine/` — SSG system → `.sc` catalogue (reverse of the
  existing importer's unit/element mapping; barycentres, rings, atmospheres and makeup all have
  known mappings because we translate them inbound already).
- **Construct marker (DECIDED: both, model optional)**: the construct always exports as a
  tiny-radius dummy catalog body carrying its exact orbital elements; an export checkbox adds a
  visible SE `Spacecraft` with a simple generic ship model instead (one .obj shipped in the
  addon — SE auto-converts to its .sm format). Either way the vista script `Follow`s the marker,
  so the camera RIDES THE CONSTRUCT'S ORBIT — terminator sweeping past, moons rising.
- **Time is 1:1 by design**: after launch SE propagates the same Keplerian system itself at
  real-time, so the vista stays truthful indefinitely with NO live link — the only motion you'd
  ever see is camera look-around and your own orbit. It only goes stale when the construct
  manoeuvres (re-export). This is why the no-API renderer is still the natural fit.
- Generated `vista.se` script: select the exported marker, `Follow`, set the in-game date to the
  SSG clock at timescale 1.0, hide UI, leave free-look on — the user looks around and zooms
  natively in SE.
- Packaged as a downloadable addon zip + a short "drop into addons/, launch with this script"
  README; button lives on the construct panel ("Vista… → Open in SpaceEngine").
- Limitations stated plainly: snapshot of the schedule (re-export after manoeuvres), ~30 s cold
  boot, other ships appear only if exported as structures (below).

**Civilisation visuals (DECIDED: structures only in V2.2; city lights banked).** SE's `Structure`
and `Spacecraft` catalog classes (0.990 dev blog; engine-supported, under-documented in the
manual) take a `Model` (.sss scheme; .obj auto-converted) and either `FixedPosPolar (lon lat
radius)` + `TidalLocked` for surface bases/starports or a normal `Orbit {}` for stations; model
materials support emissive maps that glow/bloom at night. So:

- Other constructs in the system export as orbital `Structure`/`Spacecraft` entries with a small
  set of generic models — the window shows the station you're docked at and traffic on its orbits.
- Surface starports/bases where the fiction wants them (driven by construct location/body tags).
- Emissive materials give night-side glints — "civilisation" without touching planet textures.
- **City lights are BANKED (V2.3 candidate)**: SE does them only via a `GlowMap` texture
  (`GlowMode "Night"`), which requires exporting full surface cubemap texture sets per world —
  a whole texture-export pipeline bolted onto the Part A equirect generator. Real, but not V2.2.
- SE structure rendering limits, stated for expectations: no shadows, no custom light sources,
  no reflections on structures.

**Phase B-2 (spike, then decide): Gaia Sky live bridge.**

- Exporter: SSG system → Gaia Sky JSON descriptors, mapping our physics to its procedural
  generation inputs (makeup fractions, hydrosphere/cloud cover, Teq, radius → surface/cloud/
  atmosphere seeds and palettes). This mapping is the interesting design work — our physics
  already knows what the world should look like.
- Live link: SSG (browser) → Gaia Sky REST server on localhost — camera anchored at the
  construct's propagated position, re-posed as the SSG clock scrubs; look-around either free in
  Gaia Sky or driven from an SSG "viewport frame" UI.
- Spike must verify three unknowns before committing: (a) browser-to-localhost CORS on the REST
  server (may need a tiny relay, or its Python channel instead); (b) runtime dataset loading of
  our generated descriptors; (c) whether procedural quality on OUR worlds is good enough to be
  worth it vs SE's static vista.

**Not pursuing:** Universe Sandbox (no surface at all); OpenSpace (keep on file — if Gaia Sky's
CORS/runtime loading fails, its WebSocket + JS client is the fallback channel, at the cost of the
content pipeline); Celestia (superseded: SE beats it for Tier 1 looks, Gaia Sky for Tier 2
control).

### B4. Sync model (Tier 2, for the spike)

- **Time**: SSG clock is master; push epoch + timescale on change (renderer propagates orbits
  itself between pushes — both sides use Keplerian elements, so drift is nil by construction).
- **Position**: camera anchor = construct's propagated position pushed on scrub/journey progress;
  during an engine burn the construct is between Keplerian arcs, so push at a low rate (~1-2 Hz)
  and let the renderer's camera damping smooth it.
- **Orientation**: we do not model construct attitude, so "look around" is genuinely free — no
  sync needed. Zoom = FOV set from either side.

### B5. Risks / bench-test list

- SE: behaviour of passing a `.se` script to an ALREADY-running instance is undocumented — bench
  test; assume cold boot per vista otherwise.
- SE: waypoint coordinate frames need one careful round-trip test (our plane → SE's frame),
  same class of work as the importer's element mapping.
- SE: `Follow` on a catalog `Structure`/`Spacecraft`/dummy body — scripting docs say `Select`
  takes any "space object, ship, waypoint"; bench-test that our exported marker is selectable
  and followable, and that a tiny-radius dummy body renders invisibly (no label/HUD clutter).
- SE: .obj → .sm auto-conversion for the optional ship/structure models — verify on first-load
  behaviour and cache location.
- Gaia Sky: the three spike unknowns in B3.
- Per-seat economics: SE costs money and Gaia Sky is a 1 GB+ install — reinforces "GM machine +
  table screen", not per-player.
- Scope creep guard: structures yes, planet-texture export no — city lights (GlowMap pipeline)
  stay banked until V2.3 regardless of how tempting the equirect generator makes them.

## Decisions log (review of 2026-07-08)

Part A:

1. Bodies: lit ROTATING oblate-spheroid meshes with procedural equirect textures (upgraded from
   billboards — animation economics).
2. Real 3D from day one: `propagateState3D` applies i/Ω; the 2D orrery stays untouched, and the
   resulting small azimuth divergence on inclined orbits is ACCEPTED.
3. Skins: everything per-skin — tint (cyan/green/amber/red) AND a true-colour scheme, grid mode
   (polar/square) per skin; all combinations possible via uniforms/flags.
4. Scale: toytown compression ON by default, true-scale toggle.
5. GM sync: highlight ping; if the pinged body is off-screen, auto-rotate to centre-frame it,
   then return control. Camera otherwise always free.
6. Projector mode: YES, from day one (same component, projector chrome).
7. Galaxy-level 3D: IN V2.2 scope (Phase 3).
8. Drop-stems: always on (functional height read).
9. Sound: NONE in V2.2 — BANKED as a future GENERAL Field Guide sound feature (all tiers, not a
   holo-only gimmick).

Part B:

10. Route: SE vista export ships in V2.2; Gaia Sky spike runs in parallel (its procedural
    generation was verified real after challenge — see B2); Tier 2 decided on spike evidence.
11. Deployment: GM machine + table screen accepted; GM owning SpaceEngine is an acceptable
    assumption for the vista feature.
12. `.sc` export ships as a GENERAL File > Export feature, not vista-only.
13. Construct marker: dummy body always + optional visible `Spacecraft` model (export checkbox).
14. Civilisation visuals: structures only (orbital stations + surface bases, emissive night
    glints). City lights (GlowMap texture pipeline) BANKED for V2.3.
