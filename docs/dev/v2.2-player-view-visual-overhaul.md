## WS1 — Unified planet-appearance engine

### Goal
One source of truth for a body's appearance, consumed identically by the **2D SVG disc**, the **2D orrery canvas**, and the **3D holo**. Add the missing surface phenomena (craters, irregular/asymmetric cratering, volcanism tiers, cryovolcanism/geysers, emissive glow) once, everywhere.

### Current state (grounded)
Three renderers, **one** shared seam (the apparent-colour palette). Everything else is duplicated or missing:

- **2D SVG disc** `src/lib/catalogue/PlanetDisc.svelte` — the richest renderer. Draws bands, oblate squash, toroid, small-body outline, axial tilt, terminator, polar caps, self-luminous halo, atmosphere limb-glow, **craters**, auroras, magma/lava patches, rings, belt fields. **All feature logic is inline** in the reactive block `PlanetDisc.svelte:139‑224` — no shared table.
- **2D orrery** `src/lib/components/SystemVisualizer.svelte:997‑1052` — a *third* partial reimplementation: uses `getPlanetTexture` for the surface but redraws terminator/magma itself; **no craters, caps, or auroras**.
- **3D holo** `src/lib/holo/scene.ts` — `THREE.CanvasTexture` from `getPlanetTextureEquirect()` (`scene.ts:1301`) on a lit `MeshStandardMaterial`. Draws bands/ocean/haze + auroras (separate shell) + shadows. **Does NOT draw craters, polar caps, discrete lava vents, limb-glow, or the self-luminous glow.**

The equirect texture (`planetTexture.ts:208+`) reads only the palette; it knows nothing about tags. So 3D and orrery can never reach the disc's richness without duplicating the disc's inline logic.

### Sub-issue A — Rotation / spin / orbital direction (**confirmed bug**)
Two independent defects:

1. **3D discards retrograde spin.** `rotation_period_hours` carries a sign to encode retrograde (`BodyBasicsTab.svelte:514,539,556`), but the holo does `Math.abs(node.rotation_period_hours)` at `scene.ts:1351`, so every body spins prograde in 3D.
2. **Hand-baked rings/belts pick a fixed sense with flipped chirality.** Ring/belt `omega` is always positive (`scene.ts:2381`); real moons/planets go through the propagator and honour their elements, but rings/belts don't. Separately, `positionToScene` swaps Y↔Z (`scene.ts:569‑573`), flipping handedness relative to the 2D orrery's CCW `atan2(y,x)` (`SystemVisualizer.svelte:289‑291`). Net: a top-down 3D view can spin **opposite** to the 2D orrery for the same system — the "moons/discs go the wrong way" report.

**Fix direction:** make the 2D orrery the canonical reference (prograde = CCW from north). Honour the sign of `rotation_period_hours` in 3D spin; drive ring/belt direction from the same orbital sense as the propagator; correct the chirality so 2D and 3D agree. **[Q1] confirm the 2D orrery is the reference convention** (vs. re-deriving "correct" prograde from physics and fixing both if the 2D orrery is itself wrong).

### Sub-issue B — Cratering, irregular cratering & other surface phenomena as first-class features
Today **craters are derived, not tagged**: `airless + (geology/inactive | science/impact-record | small-body)` at `PlanetDisc.svelte:170‑179`. There is **no** irregular/asymmetric-surface tag. Tidally-locked worlds should crater **differently per hemisphere** (leading vs trailing face) — not currently modelled anywhere.

Proposed appearance-driving tag vocabulary (classifier-emitted, GM-overridable, same pattern as existing tags in `tags/tagPresentation.ts`):

| Tag | Meaning | Drives |
|---|---|---|
| `surface/cratered` | heavy impact record (airless, old, busy neighbourhood) | crater field density |
| `surface/cratered-asymmetric` | tidally-locked ⇒ leading-hemisphere bias | craters weighted to one face |
| `tidal/volcanism`, `tidal/lava-flows`, `tidal/hotspots` | *(exist)* volcanism tiers | lava patches + **emissive glow** |
| `activity/cryovolcanism` | icy geysers/plumes | geyser plumes (3D) + emissive |
| `surface/irregular-terrain` | chaotic/ridged/grooved terrain | procedural terrain overlay |

"Busy neighbourhood" (lots of belts/clouds/planets ⇒ more impacts) becomes a classifier input for `surface/cratered`.

### Sub-issue C — Emissive sources
Lava, cryovolcanic plumes, self-luminous brown dwarfs, and aurorae are *glowing* and must read as light sources. In 3D only the star (point light + corona) and auroras (additive shell) are emissive today; lava/geyser/brown-dwarf glow have **no** 3D treatment (`scene.ts` has no magma/geyser emissive). Add emissive materials/sprites for these.

### Sub-issue D — Disc-gallery completeness
`routes/discgallery/+page.svelte` demonstrates craters (one), lava world, caps, oblateness/toroid, auroras, limb-glow, rings. **Missing:** `tidal/volcanism`/`tidal/hotspots` tiers, cryovolcanism/geysers, `surface/cratered-asymmetric`, belt/ring rock fields, self-luminous+aurora combos. Every feature the engine can draw must have a gallery tile — the gallery is the visual contract.

### Sub-issue E — 3D "everything is brown"
**Same colour source, different presentation.** Both 2D orrery and 3D holo start from `getNodeColor` (`colors.ts`). The 2D orrery shows the flat, fully-saturated swatch / derived `apparentColorHex`; the 3D holo pipes it through equirect texture-averaging + a single point light + low ambient (`scene.ts:381`), which mutes chroma and pushes rocky worlds toward the `terrestrial` brown default `#cc6600` (`colors.ts:36`). **[Q2] pick the 3D colour treatment** (options in the questions section).

### Proposed design — the appearance seam
Introduce a pure function:

```
deriveAppearance(body) -> AppearanceModel {
  baseColor, bands, oblateness, tilt, ringSpec, smallBodyShape,
  terminator, polarCaps, craterField {density, asymmetryAxis?},
  magmaVents[], geyserPlumes[], selfLumGlow, atmGlow, auroraOvals,
  spin {periodSec, retrograde}, ...
}
```

resolved **once** from tags + properties (lifting today's `PlanetDisc.svelte:139‑224` block into a module). Each renderer becomes a dumb consumer:
- SVG disc → draws the model in SVG,
- orrery canvas → draws the model in 2D canvas,
- holo → bakes model into the equirect texture + builds THREE meshes/sprites for 3D-only features (geysers, emissive shells).

This is the "same engine" the idea asked for: not one renderer, but one **appearance model** with three thin renderers. **[Q3] scope: full seam now vs. incremental parity first.**

**Principle — unify the MODEL, not the pixels (Alex, 2026-07-18).** The model decides *which* features a body shows and *how strong* (colour/tier/count/strength); each renderer draws them in its own best form and **need not match pixel-for-pixel**. The model returns FLAGS + PARAMS, never SVG paths (already the case).

- **Core surface ≈ the same** in 2D and 3D (the base textured/coloured sphere). The 3D differentiator is **glow + emission**, only where 3D is genuinely better — otherwise don't bother diverging. A modest 3D-only nicety: **sea glisten** (specular highlight on liquid).
- **3D-only emissive enhancements (the payoff), driven by the model's params:**
  - **Auroras** — already better in 3D (emissive additive shell); colour + scale vary with strength. Keep as-is; do NOT port the 2D oval down.
  - **Volcanoes / lava** — **glow** (emissive vents), scaled by the magma tier. 3D currently has none (equirect only flat-tints) — this is a real gain.
  - **Cryovolcanic plumes (Enceladus)** — **water/ice jets that squirt outward and glisten**, pushing far out into space, **scaled to the body's current rendered radius/view**. A 3D-only particle/emissive feature with no 2D equivalent — highest-wow item.
  - **Self-luminous brown-dwarf glow** — emissive halo (3D currently missing it).
  - **Black hole — "Interstellar" treatment (Alex, 2026-07-18).** Two upgrades on the existing BH handling (`scene.ts:1198`, `bhFeeding` `:1825‑1831`, blue-white accretion glow sprite `:1211‑1217`):
    - **Temperature-graded accretion disc** — the disc (currently a generic icy particle ring) should GLOW as a real accretion disc: hot-graded by radius — deep **red/orange at the outer edge → yellow → white-hot at the inner edge** (ISCO), emissive/additive, brightness scaled by the Eddington fraction (`accretionEddington`). Replace the flat blue-white glow with this gradient. This is the tractable, high-impact part (similar shape to the volcano-vent work).
    - **Screen-space gravitational lensing shader** — the "fast Interstellar vibe": a stylized `ShaderPass` in the holo `EffectComposer` (same chain as the CRT filters) that warps the background around the hole (analytic deflection ∝ 1/impact-param) + an Einstein ring, fed the BH's screen position + radius. This is the banked **§A13** plan in `docs/dev/v2.2-3d-design.md` — do NOT full-GR ray-march (desktop-only); the stylized version is browser/mobile-cheap. Drives off `star/BH`·`star/BH_active`·`accretionEddington`·Schwarzschild radius.
- The model guarantees 2D and 3D agree on *what appears and how strong* without flattening 3D to the 2D look.

### Tasks (once decisions land)
1. **DONE @v2.1.148-beta** — Extract `deriveAppearance()` from `PlanetDisc.svelte` into `src/lib/rendering/planetAppearance.ts`; SVG disc consumes it (no behaviour change, /discgallery regression-checked; 7 tests).
2. **DONE @v2.1.147-beta** — Fix rotation: holo spin negated to co-rotate with orbits/rings + signed `spinPeriodSec` for retrograde (`scene.ts` `updateSpin`).
3. Add tags (`surface/cratered`, `-asymmetric`, `activity/cryovolcanism`, `surface/irregular-terrain`) to classifier + `tagPresentation.ts` + editor.
4. Route orrery canvas + holo equirect through `deriveAppearance()`; add craters/caps/limb-glow to both.
5. Add 3D-only emissive features: **lava/volcano glow DONE @v2.1.149-beta** (`buildMagmaVents`/`updateMagma` in `scene.ts`, flickering hot-spot sprites per the model's magma tier); STILL TODO: **cryovolcanic geyser plumes** (Enceladus jets — the high-wow item), **brown-dwarf self-luminous glow**, **black-hole temperature-graded accretion disc** (red→yellow→white-hot) + banked **§A13 lensing shader** (Interstellar vibe), sea glisten.
6. Fill disc-gallery tiles for every feature.
7. Resolve 3D colour treatment per **[Q2]**.

## WS2 — "Text" system view rebuilt as an interactive Guide document

### Goal
Replace the weak current "Text" system view with the old *The Guide* look/feel: a small **interactive document** — object **hierarchy** to navigate, the **GM-selected images**, an info block that is **part of the page** (slide-in but in-document, not chrome), plus reinstated **simple system line-drawings**. It renders as a low-fps document that is **snapshotted to an image and filtered** at the last moment.

### Current state (grounded)
- Legacy schematic browser `CatalogueBrowser.svelte` still exists: an **SVG log-scale orbital line-diagram** (`:194‑222`) + a chip hierarchy picker (`:225‑238`) + an **inline** info panel (`:250‑316`) showing imagery three ways (`disc`/`photo`/`none`). **This is essentially the old Guide** and it still works on legacy skins.
- But the **new preset system routes `diagram2d` away from SVG**: `effectiveSystemTier` maps `diagram2d → 'holo'` (`catalogue/+page.svelte:542‑543`) and renders a **flattened 3D holo**, not the schematic. So from the new player-view path, the "simple drawings" and the Guide's interactive schematic were effectively removed.
- The Guide itself is **not gone** — it's `THEMES.guide` (legacy) and `BUILTIN_PRESETS[0]` id `guide` (`presets.ts:97‑102`), still the default. What's masked in `CompanionModal.svelte:48‑49` is the **Holo Table**, not the Guide.
- Filter pipeline **already supports snapshot-under-filter**: `FilteredListView.svelte` draws the list to an offscreen canvas and only re-renders on change, then composites through a **real GPU shader** (`filteredCanvas.ts`), with tap-mapping through the warp (`warpPoint`). Cover + list already do canvas→shader.
- Headers/footers: the preset path **already draws `companyName`/`footerText`/`guideTips` INTO the canvas** (`coverCard.ts:101‑105`, `drawList`, `drawHud`) — i.e. inside the filtered document, exactly the target. Only the device status-bar remains outside chrome.

So the pieces exist; they're just not wired into a single "Guide document" module.

### Proposed design
Add a player-view **system module** (working name `document` / `guide`) that renders the Guide as a **canvas-drawn interactive document**, reusing the proven `drawList`/`drawCover`→shader pipeline:

1. **Schematic** — port `CatalogueBrowser`'s SVG orbital line-diagram into a canvas draw (`drawSystemSchematic`) so it goes through the shader. This reinstates the "simple drawings."
2. **Hierarchy navigator** — the chip/tree picker, canvas-drawn, tap-to-select via a hit-map (like `FilteredListView`'s tap mapping).
3. **Imagery** — GM-selected `image.url` (photo) or procedural `PlanetDisc` (disc) per preset `bodyGfx`, drawn into the document.
4. **In-document info block** — the slide-in panel becomes a canvas region within the filtered document (not a DOM aside), matching the old Guide.
5. **Low-fps snapshot** — re-draw only on data/selection/filter change; the shader animates only time-based uniforms. The idea's "image at the last moment before the filter" is this model.
6. **Headers/footers inside the page** — reuse `guideTips`/footer-into-canvas; migrate the remaining status-bar affordances into the document where the design calls for it.

**[Q4] interactivity vs. pure snapshot.** A canvas document needs a manual hit-map for taps (doable, as `FilteredListView` proves) — but if a preset is non-interactive (`interactive:false`), it's a pure filtered image. Confirm we want the **interactive** canvas-document (hit-map) as the default Text/Guide, with non-interactive as a flag.

### 7. ONE player-info render pipeline, reused across ALL views (incl. 3D) — added 2026-07-18
Observed while testing WS4 on beta v2.1.147: over the **3D holo** view, the **info panel appears in the wrong place (overlaying the middle of the image)** and the **Guide headers/footers render mangled**. Root cause is the split established elsewhere in this doc — the info panel and the header/footer chrome over the 3D view are still **DOM with a CSS-approx filter** (`.inspector` aside + `cssFilterApprox`, `catalogue/+page.svelte:1183`, `bd00d8a`), positioned in screen space and NOT following the warped/insetted 3D projection, while the cover/list already draw their chrome INTO the canvas (`coverCard.ts`, `drawList`, `drawHud`). So the same "player info screen" is rendered two different ways depending on the view, and the DOM way misplaces under the 3D warp + info-panel inset (`viewInsetRight`, `holo/scene.ts` `setViewInset`).

**Decision (Alex, 2026-07-18): unify these onto ONE render pipeline.** The canvas-draw→shader pipeline WS2 builds for the Guide document (info block, headers/footers, imagery) must be the SAME pipeline used to draw the info panel and header/footer chrome **over the 3D holo view** — drawn into the scene's HUD quad (`drawHud` is the seam) so it warps and insets WITH the 3D content instead of floating as mis-positioned DOM. Net goal: a player "info screen" (cover, document, per-body info block, headers/footers) is drawn ONE way — canvas region → shader — regardless of whether the underlying view is the 2D document, the 2D orrery, or the 3D holo. Retire the DOM `.inspector` + `cssFilterApprox` chrome for player views; keep DOM only where a preset is explicitly un-filtered.

Design implication for WS2: build the document renderer as a **reusable `drawInfoChrome(ctx, {body, header, footer, theme})`** primitive (not baked into the document module), so the holo path calls the identical primitive into its HUD quad. This is the "simplify to a consistent render pipeline for these player info screens" ask — do it as part of WS2 so WS2's module and the 3D HUD share code from day one.

### 8. Document STYLE range — friendly book ↔ formal terminal report (Alex, 2026-07-18)
The document must be **one renderer, many looks**, spanning from a warm illustrated *field-guide book* to a *formal green-screen text-terminal report* — the whole tonal range, driven entirely by the preset **theme** (nothing hard-coded). Requirements:
- **Font + colour fully passed through** from the preset theme into every text draw (heading/body/label/list), not just the accent. A book preset picks a serif + parchment palette; a terminal preset picks a mono + phosphor-green palette; same content, same code.
- **A small library of LIST STYLES to suit genre**, selectable per preset — e.g. *illustrated bullets* (book), *numbered dossier* (report), *terminal log* (`>` prefixed, monospace, blinking caret), *ledger/table* rows, *ticker/manifest*. Each is a `drawList`-style routine over the same data; the preset names which.
- **Built to make the filter shine.** Because the document is canvas→shader (§5/§7), the styles should be chosen so the CRT/scanline/phosphor/warp filters make them "come alive" — a terminal-log list under the green CRT filter should read like a real ship console; the book list under a soft/none filter reads like a printed page. Design the styles WITH their target filters in mind.
- Ties into the theme model already in `presetTypes.ts` (font stack + accent + filter) — extend it with a `documentStyle` / `listStyle` enum rather than a new system.

### Tasks
1. New `document` system module + `drawSystemSchematic` (port from `CatalogueBrowser:194‑222`).
2. Canvas hierarchy navigator + hit-map.
3. In-document info block (canvas region) + imagery.
4. Wire `diagram2d`/a new `document` preset value to it (decide whether it **replaces** the flattened-holo `diagram2d` or is a new module — see **[Q4]**).
5. Confirm filter snapshot cadence + header/footer-in-page.
6. **Reusable `drawInfoChrome()` primitive** shared by the document module AND the 3D holo HUD (`drawHud`), so the info panel + headers/footers draw the SAME way over 2D and 3D and stop mis-positioning under the 3D warp/inset. Retire the DOM `.inspector` + `cssFilterApprox` for filtered player views.

## WS3 — Unified grid/overlay across all views

### Goal
Every spatial view (2D system, 3D system/holo, 2D starmap, 3D starmap) offers the **same** overlay menu: **None · Square · Hex · Traveller Hex · Polar · Polar+Scale**, all correctly scaled. Traveller Hex is available to **all** users, decoupled from Traveller *mode*. Route/starlink **names** respect the Hide-labels checkbox.

### Current state (grounded)
Two disjoint "grid" concepts:
- **Decorative (3D engine):** `GridMode = 'off'|'plain'|'scaled'|'hex'` (`starmapScene.ts:22`, `holo/scene.ts:259`, `presetTypes.ts:98`); UI `PlayerPresetEditor.svelte:230‑237`. No square, no traveller-hex.
- **Snap grid (2D SVG editor):** `GridType = 'grid'|'hex'|'none'` (`starmapUiStore.ts:8`); `Grid.svelte` also implements a full **`traveller-hex`** renderer (numbered CCRR hexes, subsector zig-zags, `Grid.svelte:68‑157`) but it's **never offered** in the dropdown — it's synthesized only when Traveller mode is on **and** grid=hex (`Starmap.svelte:288‑294`).
- **2D system view (`SystemVisualizer`) has NO grid at all.**
- Traveller-hex numbering/subsectors exist **only** in the 2D SVG editor; the 3D engine draws only the hex lattice, no numbers.

### Overlaps / decoupling
The idea's "Traveller Hex as a normal dropdown option for all users" is structurally easy: `Grid.svelte` already accepts `'traveller-hex'` as a plain prop. Re-add it to `GridType` (`starmapUiStore.ts:8`) + the select (`SettingsModal.svelte:309‑313`), and decouple from `travellerMode` at `Starmap.svelte:288‑294` and `+page.svelte:624`. Traveller **mode** (parsec scaling, import placement, snapping) stays a separate flag — it just defaults the grid to Traveller Hex.

### Proposed design
1. **One overlay enum** shared by all views: `none | square | hex | travellerHex | polar | polarScaled`. Reconcile the two current enums onto it.
2. **2D system view gains grids** — add square/hex/traveller-hex/polar/polar-scaled rendering to `SystemVisualizer` (currently none).
3. **Traveller Hex numbering in 3D** — port the CCRR numbering + subsector lines from `Grid.svelte:68‑157` so 3D isn't a lattice-only downgrade (or accept lattice-only in 3D — **[Q5]**).
4. **Traveller Hex for all users** — dropdown option, decoupled from mode.
5. **"Polar scale" everywhere** — expose the 3D `scaled` polar on the system + starmap dropdowns (idea: "like the new 3D system uses by default").
6. **Route names respect Hide-labels** — route names render unconditionally in the 2D editor (`Starmap.svelte:1050‑1064`) and not at all in 3D (`SmRoute` has no name, `starmapScene.ts:21`). Add a `name` to 3D routes, render it, and bind route-name visibility to the same `labelsHidden` override that governs system/body labels (`starmapScene.ts:318,463`).

### Risk
Scaling correctness across polar-scaled + toytown compression + traveller parsec units is the fiddly part; each overlay must read the same world→scene scale the bodies use.

## WS4 — Click accuracy under the CRT warp (bug)

### Goal
A click lands on the object under the pixel the user aimed at, even when the view is skewed/rolled/scaled by the CRT filter.

### Root cause (grounded)
The hit-test is computed in **un-transformed** canvas space, while an **ancestor CSS transform** shears/scales the displayed pixels:
- Companion CRT skin: `transform: skewX(...)` on `<main>` (`catalogue/+page.svelte:100‑101`), which contains the interactive `SystemVisualizer`.
- Projector: `transform: scale(0.98)` (`projector/+page.svelte:210‑215`).

`gestures.ts:48‑51` computes `clientX - rect.left` where `rect` is the **transformed** bounding box, and `screenToWorld` (`SystemVisualizer.svelte:424‑430`) assumes an axis-aligned canvas centred at `width/2`. Neither inverts the skew/scale. Under skewX the horizontal error grows with vertical distance from centre → "I clicked left, it picked something right," worst at top/bottom edges. The main editor has no such transform, so the bug is CRT/companion/projector-only. There is **no** literal horizontal flip and **no** `panelPadPx` on the 2D path — it's purely the un-inverted ancestor transform.

### Proposed design — two options
- **(A) Move the geometric warp off the interactive layer.** Keep skew/scale on a non-interactive overlay; leave the interactive canvas axis-aligned. Simplest, but the *pixels* the player sees are then un-warped where they interact (may fight the intended CRT look).
- **(B) Invert the ancestor transform in pointer→canvas mapping.** In `gestures.ts`, map `clientX/clientY` through the inverse of the element's full transform matrix (`DOMMatrix` / `getScreenCTM`-style) instead of `clientX - rect.left`. Keeps the warp visible **and** aligns hit-areas. Preferred — it's what "roll the click areas with the gfx" means.

For the **GPU-shader** paths (holo/list) the warp is in the shader; `warpPoint` (`filteredCanvas.ts:69‑80`) already inverts it for taps — so the fix is specifically the **CSS-transform** paths around `SystemVisualizer`.

**Decision: option B (invert transform), behind a swap-back seam.** The inverse-matrix mapping is unproven in practice ("sometimes it works, sometimes not"), so implement it so option A is a **one-flag fallback**: put the pointer→canvas mapping behind a single helper (e.g. `mapPointerToCanvas(el, clientX, clientY, {invertTransform})`) and gate the inversion on a config/preset flag defaulting on. If it misbehaves we flip the flag to fall back to moving the warp off the interactive layer — no code surgery.

## WS5 — Player-view lock-down

### Goal
GM-controlled confinement:
1. **No starmap supported ⇒** GM picks one system; the player is dropped into it and **cannot exit to the starmap**.
2. **Player can never return to a cover** (cover is a one-time intro).
3. **Cover + one selected system ⇒** that single system is all the player ever sees.

### Current state (grounded)
- `starmapEnabled: false` already hides the back button (`catalogue/+page.svelte:799`), blocks `onPopState` returning to starmap (`:611`), and auto-drops into a system (`:572‑574`) — **but the auto-pick is hardcoded to `starmap.systems[0]`** (`:573`); there is **no `pinnedSystemId`**.
- Cover shows once (`coverDismissed`, `:567`) and re-arms only on preset-id change (`:569`); there is **no** back-to-cover navigation already. So "never return to cover" is largely satisfied; just verify no path re-triggers it.
- `interactive:false` → `pointer-events:none` (`:1183`); `followGM` mirrors GM.

### Proposed design
1. Add `pinnedSystemId?: string` to `PlayerPreset` (`presetTypes.ts`); when set (or when `starmapEnabled:false`), the player is locked to that system at `catalogue/+page.svelte:573` instead of `systems[0]`. **[Q7] source of the pin** — stored in the preset (GM picks at authoring time) vs. "follow the GM's current system" live.
2. With `starmapEnabled:false`: confirm every exit path (back button `:799`, `onPopState` `:611`, any deep link) refuses to surface the starmap.
3. Verify no navigation re-shows the cover once dismissed; if a "return to cover" affordance exists anywhere, gate it off for players.
4. Cover + pinned system compose: cover intro → drop into the one system → no starmap, no cover-return.

### Risk
Low — mostly gating existing flags + one new field. Main care: deep links (`?preset=`, `?sid=`) must not bypass the lock.
